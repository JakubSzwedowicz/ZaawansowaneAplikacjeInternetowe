UŚCIŚLENIE (dodano 13.10.2025): W trakcie pierwszych konsultacji pojawiły się wątpliwości o znaczenie określenia "poziom dojrzałości 2.3 dla REST API". Generalnie intencją tutaj jest to, aby zaimplementowali Państwo API w swoich aplikacjach na poziomie drugim według modelu dojrzałości zaproponowanym przez Leonarda Richardsona, tj. z uwzględnieniem szczegółów takich jak:

stosowanie metod protokołu HTTP właściwych dla akcji podejmowanych na zasobach udostępnianych przez API (np. używamy metody DELETE dla usuwania zasobu, a GET do jego pobierania),
wykorzystanie nagłówków protokołu HTTP dla uzgodnienia formatu treści (np. Content-Type, Accept),
wykorzystanie parametrów żądania dla operacji takich jak filtrowanie zasobów, sortowanie itp.,
stosowanie kodów odpowiedzi HTTP stosownych do znaczenia odpowiedzi wysyłanej przez serwer (np. klasyczny błąd to wysłanie kodu odpowiedzi HTTP 200 OK, a w ciele odpowiedzi komunikatu o błędzie...).
Więcej informacji na ten temat znajdziecie Państwo w poniższych artykułach:

https://martinfowler.com/articles/richardsonMaturityModel.html
https://www.geeksforgeeks.org/node-js/richardson-maturity-model-restful-api/
https://learn.microsoft.com/pl-pl/azure/architecture/best-practices/api-design
Nie jest wymagana implementacja API na poziomie trzecim dojrzałości, czyli tzw. HATEOAS (Hypertext As The Engine Of Application State).
